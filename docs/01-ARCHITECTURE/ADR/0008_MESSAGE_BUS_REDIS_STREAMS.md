# ADR 0008: Use Redis Streams for the message bus with deduplication

## Status

**Proposed**

## Context

Block 3 (Nervous System) requires a message bus for event distribution between Office and Factory components. The message bus must:

1. **Reliably deliver events** from producers (Office connectors, Intent Analyzer, Session Manager) to consumers (Factory, ZenJournal, KB Ingestion).
2. **Support multiple consumers** with consumer groups for parallel processing.
3. **Provide at-least-once delivery** with deduplication to handle retries and network issues.
4. **Integrate with existing infrastructure** – Redis is already required for Tier 1 (Hot) memory cache.
5. **Scale from local development** (single-node Redis) to production (Redis Cluster) without architecture changes.
6. **Support event deduplication** using `zen-sdk/pkg/dedup` as mandated by the construction plan.

Alternatives considered: NATS (lightweight but adds another dependency), Kafka (heavyweight, overkill), in‑process channels (not durable, no cross‑pod communication).

## Decision

**Use Redis Streams as the message bus backbone**, with a generic `MessageBus` interface that abstracts the underlying implementation. The interface will be defined in `pkg/messagebus/interface.go`, with a Redis Streams implementation in `internal/messagebus/redis`.

### Interface Design

```go
// MessageBus defines the generic pub/sub interface for event distribution.
type MessageBus interface {
    // Publish publishes an event to a stream.
    Publish(ctx context.Context, stream string, event *Event) error

    // Subscribe creates a subscription to a stream.
    // If group is empty, a unique consumer group is created.
    Subscribe(ctx context.Context, stream, group string) (Subscription, error)

    // CreateConsumerGroup creates or ensures a consumer group exists.
    CreateConsumerGroup(ctx context.Context, stream, group string) error

    // Ack acknowledges successful processing of an event.
    Ack(ctx context.Context, stream, group, id string) error

    // Close closes the connection.
    Close() error
}

// Event represents a message bus event.
type Event struct {
    ID          string                 // Stream entry ID (auto‑generated by Redis)
    Type        string                 // Event type (e.g., "work_item.created")
    Source      string                 // Producer identifier
    Correlation string                 // Correlation ID for tracing
    Payload     []byte                 // Encoded payload (JSON, Protobuf, etc.)
    Metadata    map[string]string      // Additional headers
    Timestamp   time.Time              // Event creation time
}

// Subscription represents an active subscription.
type Subscription interface {
    // Events returns a channel of incoming events.
    Events() <-chan *Event

    // Errors returns a channel of subscription errors.
    Errors() <-chan error

    // Close closes the subscription.
    Close() error
}
```

### Deduplication Integration

The `MessageBus` implementation will integrate `zen-sdk/pkg/dedup` at the producer side (optional) and consumer side (mandatory for idempotent processing). Deduplication keys will be derived from `Event.Source` + `Event.Correlation` + hash of `Event.Payload`.

### Configuration

- `REDIS_URL` environment variable (default: `redis://localhost:6379`)
- `MESSAGEBUS_DEDUP_WINDOW_SECONDS` per source (default: 60 seconds)
- `MESSAGEBUS_MAX_PENDING` per consumer group (default: 1000)

### Stream Naming Convention

Streams follow a hierarchical pattern:

- `office:work_item` – Work item events from Office connectors
- `office:intent` – Intent analysis events
- `factory:task` – Factory task events
- `journal:event` – ZenJournal events
- `kb:ingestion` – KB ingestion events

### Consumer Group Naming

- `zen-office` – Office connector consumers
- `zen-intent-analyzer` – Intent Analyzer consumers
- `zen-factory` – Factory consumers
- `zen-journal` – ZenJournal writer consumers

### Redis Streams Advantages

1. **Durability** – Events are persisted and survive Redis restarts (if configured with AOF).
2. **Consumer Groups** – Built‑in support for at‑least‑once delivery with explicit acknowledgment.
3. **Automatic ID generation** – Timestamp‑based IDs provide natural ordering.
4. **Range queries** – Ability to replay historical events.
5. **No additional infrastructure** – Uses existing Redis instance.

### Implementation Steps

1. Define `pkg/messagebus` interface.
2. Implement `internal/messagebus/redis` with Redis Streams.
3. Add deduplication wrapper using `zen-sdk/pkg/dedup`.
4. Create integration tests with a test Redis container.
5. Update `internal/office/jira` connector to publish events via the message bus (optional, can be phased).
6. Update `internal/factory` to subscribe to `office:work_item` stream.

## Consequences

### Positive

- **Unified event distribution** – All components use the same bus, enabling observability and tracing.
- **Durability** – Events are not lost if a consumer crashes.
- **Scalability** – Multiple consumers can share load via consumer groups.
- **Deduplication** – Prevents duplicate processing of the same event.
- **No new infrastructure** – Leverages existing Redis deployment.
- **Aligned with construction plan** – Uses `zen-sdk/pkg/dedup` as required.

### Negative

- **Redis becomes a single point of failure** – Can be mitigated with Redis Sentinel or Cluster in production.
- **Additional complexity** – Need to manage consumer group offsets and pending messages.
- **Latency overhead** – Network round‑trip to Redis vs in‑memory channels.

### Neutral

- Events are encoded as JSON (default) but the interface supports other encodings.
- The interface allows swapping Redis for another backend later (e.g., NATS) without changing component code.

## Alternatives Considered

### 1. NATS

- **Pros**: Extremely fast, built‑in clustering, supports both at‑most‑once and at‑least‑once semantics.
- **Cons**: Adds another infrastructure component, not already in the stack, operational overhead.

### 2. Apache Kafka

- **Pros**: Industry‑standard, extremely durable, excellent ordering guarantees.
- **Cons**: Heavyweight, complex to operate, overkill for Zen‑Brain’s expected volume.

### 3. In‑process Go channels with persistence to ZenJournal

- **Pros**: No network overhead, simple.
- **Cons**: Not durable across pod restarts, cannot communicate across pods, requires a separate synchronization mechanism.

### 4. Redis Pub/Sub (not Streams)

- **Pros**: Simple, low latency.
- **Cons**: No persistence, no consumer groups, at‑most‑once delivery.

Redis Streams provides the right balance of durability, ordering, and operational simplicity.
